Architecture Overview
AlloyDB is Google's fully managed PostgreSQL-compatible database built for demanding OLTP + HTAP workloads. Understanding its unique architecture is the DBA's starting point.

What Makes AlloyDB Different

AlloyDB separates compute from storage — the database engine (PostgreSQL-compatible) runs in GCE VMs, while storage is handled by a distributed, log-structured system across Google's infrastructure. This is fundamentally different from vanilla PostgreSQL.

🧠
Adaptive Query Processing
ML-driven optimizer learns query patterns, auto-tunes vacuum, and intelligently routes reads between row and columnar engines without manual hints.
⚡
Distributed Storage Layer
6-way replication across 2 zones. No shared storage bottleneck. WAL writes are durable before the engine returns ACK. 99.99% SLA for regional clusters.
📐
Columnar Engine (built-in)
Auto-converts frequently scanned columns to columnar format in memory. Delivers up to 100x speedup on analytical queries without ETL to a warehouse.
🔁
Zero-Downtime Failover
Standby instance with RPO near 0. Failover in <60 seconds. Read pool instances independently scalable and serve stale reads during primary failover.

Storage & I/O Engine
AlloyDB's distributed storage fundamentally changes how DBAs think about I/O, WAL, and durability.

How the Storage Layer Works

Unlike traditional PG where the engine writes heap pages to local disk, AlloyDB sends WAL records to the storage layer, which reconstructs pages on demand. The engine never writes full pages to storage — only log records. This is called log-is-the-database.

📝
Write Path
Engine → WAL records → Storage shard leaders → 6-way replication ACK → commit returns. Full pages reconstructed lazily by storage nodes from log replay.
📖
Read Path
Engine reads from local buffer cache first. On miss, fetches page from nearest storage shard. Storage nodes serve the latest page version from their log replay.

Key Implications for DBAs

  • No tune needed for checkpoint_segments — AlloyDB handles WAL recycling differently; there are no local WAL files to fill up.
  • wal_level = logical is supported for logical replication downstream.
  • pg_basebackup does not work — use built-in backup APIs or PITR.
  • Tablespaces — not meaningfully applicable; storage sharding is automatic.
  • Storage auto-grows — no pre-allocated disk size; billed per GB used.
  • IOPS are not a tuning knob — the storage layer is shared, elastic, and managed.

Database and table sizes
-- Top 10 largest tables SELECT schemaname, tablename, pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size, pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size, pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename) - pg_relation_size(schemaname||'.'||tablename)) AS indexes_size FROM information_schema.tables WHERE table_schema NOT IN ('pg_catalog', 'information_schema') ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC LIMIT 10;
Users, Roles & IAM Security
AlloyDB integrates both native PostgreSQL roles and Google Cloud IAM for authentication. Understanding both layers is essential.

Authentication Modes

🔑

Standard PostgreSQL username/password. Managed via CREATE USER / ALTER USER. Used by the 'postgres' superuser and internal service accounts.

Backup & Recovery
AlloyDB provides automated continuous backups and PITR. As a DBA, you must know how to validate, trigger, and restore from them.

Backup Types

📸
Automated Backups
Daily snapshots, configurable window. Retained 1–35 days. Stored in Google Cloud Storage across regions. Incremental internally.
⏱️
PITR (Continuous)
WAL archived continuously. Restore to any second within the backup window. Minimum WAL retention is 14 days by default.
📦
On-Demand Backups
Trigger manually before major changes (migrations, schema changes). Not subject to automatic expiry — you control retention.

Managing Backups

Backup operations via gcloud
# Create on-demand backup gcloud alloydb backups create pre-migration-backup \ --cluster=prod-cluster \ --region=us-central1 \ --description="Before schema migration v2.5" # List backups gcloud alloydb backups list \ --filter="clusterName:prod-cluster" \ --region=us-central1 \ --format="table(name, state, createTime, sizeBytes)" # Restore to a new cluster from backup gcloud alloydb clusters restore prod-cluster-restored \ --region=us-central1 \ --backup=projects/PROJECT/locations/us-central1/backups/pre-migration-backup \ --network=projects/PROJECT/global/networks/my-vpc # PITR restore (restore to specific timestamp) gcloud alloydb clusters restore prod-pitr-restore \ --region=us-central1 \ --source-cluster=prod-cluster \ --point-in-time="2024-11-15T14:30:00Z" \ --network=projects/PROJECT/global/networks/my-vpc
💡
Restores always create a new cluster — they do not overwrite the source. To "replace" a cluster, restore to new → validate → update DNS / connection strings → decommission old cluster.

Backup Validation Checklist

Schedule monthly DR drills

Actually restore a backup to a temporary cluster and validate data integrity. Don't assume backups work.

Validate row counts on key tables

After restore, run COUNT(*) on business-critical tables and compare against pre-restore snapshots.

Test PITR to 5 minutes before an event

Simulate an accidental DROP TABLE; restore to T-5min; confirm the table is present.

Document your RTO

Time the full restore operation. For large databases (1TB+), restores can take 30–90 min. Factor this into your SLA.

Index Strategy

Finding missing indexes and redundant indexes
-- Tables with sequential scans (missing index candidates) SELECT schemaname, relname, seq_scan, seq_tup_read, idx_scan, ROUND(seq_scan::numeric / (seq_scan + idx_scan + 1), 2) AS seq_ratio FROM pg_stat_user_tables WHERE seq_scan > 1000 AND (seq_scan + idx_scan) > 0 ORDER BY seq_tup_read DESC LIMIT 20; -- Unused indexes (consuming write overhead + storage) SELECT schemaname, tablename, indexname, idx_scan, pg_size_pretty(pg_relation_size(indexrelid)) AS index_size FROM pg_stat_user_indexes WHERE idx_scan = 0 AND indexname NOT LIKE '%pkey%' ORDER BY pg_relation_size(indexrelid) DESC; -- Create index concurrently (no table lock!) CREATE INDEX CONCURRENTLY idx_orders_user_created ON orders (user_id, created_at DESC) WHERE status != 'cancelled'; -- partial index

Query Optimization
Finding and fixing slow queries is the core DBA skill. AlloyDB's pg_stat_statements integration makes this systematic.

Identifying Slow Queries

pg_stat_statements — top resource consumers
-- Enable (add to database flags): pg_stat_statements.track = all CREATE EXTENSION IF NOT EXISTS pg_stat_statements; -- Top 10 queries by total time SELECT LEFT(query, 80) AS query_snippet, calls, ROUND(total_exec_time::numeric, 2) AS total_ms, ROUND(mean_exec_time::numeric, 2) AS avg_ms, ROUND(stddev_exec_time::numeric, 2) AS stddev_ms, rows, ROUND((100 * total_exec_time / SUM(total_exec_time) OVER ())::numeric, 2) AS pct_total FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10; -- Highest avg latency (worst individual queries) SELECT LEFT(query, 80) AS query, calls, ROUND(mean_exec_time::numeric, 2) AS avg_ms, ROUND(max_exec_time::numeric, 2) AS max_ms FROM pg_stat_statements WHERE calls > 100 ORDER BY max_exec_time DESC LIMIT 15; -- Reset stats (do periodically or after tuning) SELECT pg_stat_statements_reset();

AlloyDB Columnar Engine
The AlloyDB columnar engine is one of its most powerful differentiators — providing HTAP capabilities without a separate analytics database.

How It Works

The columnar engine maintains a columnar representation of frequently accessed data in memory (separate from the shared_buffers heap cache). The query optimizer automatically routes scan-heavy query parts to the columnar engine while OLTP operations use the row engine.

✅
The columnar engine does not require schema changes, separate tables, or query hints. It works transparently alongside your existing PostgreSQL schema.

Setup & Management

Enable and configure the columnar engine
-- Enable the extension CREATE EXTENSION IF NOT EXISTS google_columnar_engine; -- Check columnar engine status SELECT * FROM g_columnar_recommended_columns LIMIT 20; -- Manually add a table to columnar (auto-selection also works) SELECT g_columnar_columns_add_relation('public', 'orders'); -- Add specific columns (to save memory) SELECT g_columnar_columns_add('public', 'orders', 'ARRAY[''created_at'',''status'',''amount'',''user_id'']'); -- Check what's in the columnar store SELECT * FROM g_columnar_columns; -- Verify query uses columnar (look for "Columnar Scan" in plan) EXPLAIN SELECT status, SUM(amount) FROM orders WHERE created_at > now() - interval '90 days' GROUP BY status;

When Columnar Helps

✅
Great Fit
Aggregations over date ranges · Dashboards with GROUP BY · Reports scanning millions of rows · Historical data analysis · Rarely-updated wide tables
❌
Not Suitable
Point lookups by primary key · High-frequency INSERT/UPDATE tables · Very small tables · Queries fetching most columns per row

Connection Management
Connection management is critical in AlloyDB. Each backend process costs ~5MB RAM. Unmanaged connections cause OOM, lock pile-ups, and degraded performance.

AlloyDB Built-in Connection Pooling

AlloyDB includes a built-in pgBouncer-based connection pooler accessible via a separate port (typically 5433). This provides transaction-mode pooling without deploying your own pgBouncer.

Connect through built-in pooler
# Via Auth Proxy with pooler flag ./alloydb-auth-proxy \ "projects/PROJECT/locations/REGION/clusters/CLUSTER/instances/INSTANCE" \ --use-builtin-pooler # Direct pooler port (if using PSC) psql "host=10.x.x.x port=5433 dbname=mydb user=postgres"

Monitoring Connections

Connection health dashboard queries
-- Connection count by state and user SELECT usename, application_name, state, COUNT(*) AS connections, COUNT(*) FILTER (WHERE wait_event IS NOT NULL) AS waiting FROM pg_stat_activity GROUP BY usename, application_name, state ORDER BY connections DESC; -- Connections vs max_connections SELECT COUNT(*) AS current_connections, (SELECT setting::int FROM pg_settings WHERE name='max_connections') AS max_connections, ROUND(100.0 * COUNT(*) / (SELECT setting::int FROM pg_settings WHERE name='max_connections'), 1) AS pct_used FROM pg_stat_activity; -- Idle connections holding locks (a bad pattern) SELECT pid, usename, state, now() - state_change AS idle_for FROM pg_stat_activity WHERE state = 'idle in transaction' ORDER BY idle_for DESC;
🚨
"idle in transaction" connections are the #1 lock hazard. Set idle_in_transaction_session_timeout = 300000 (5 min) as a cluster flag to auto-terminate these.

Troubleshooting
Systematic approaches to diagnosing AlloyDB performance incidents, connection storms, and resource exhaustion.

Incident Response Framework

Triage: What's broken?

Check Cloud Console alerts, pg_stat_activity for blocked queries, CPU/memory metrics. Is the primary up? Are read pools lagging?

Isolate: Which layer?

App → Connection pool → Database → Storage. Check error messages, log files in Cloud Logging, and pg_stat_activity wait events.

Contain: Stop the bleeding

Kill blocking queries, redirect reads to read pool, scale instance if OOM, or roll back recent deployments.

Resolve: Fix root cause

Add missing index, fix query, tune GUC, increase pooling limits, or scale instance. Document and create alerts.

High CPU — Diagnosis Script

Full CPU diagnosis playbook
-- Step 1: What queries are consuming CPU? SELECT pid, usename, state, now() - query_start AS runtime, wait_event, LEFT(query, 120) AS query FROM pg_stat_activity WHERE state = 'active' ORDER BY runtime DESC; -- Step 2: Historical top CPU from pg_stat_statements SELECT LEFT(query,80), calls, total_exec_time, mean_exec_time FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 5; -- Step 3: Are sequential scans killing us? SELECT relname, seq_scan, idx_scan, ROUND(100.0*seq_scan/(seq_scan+idx_scan+1),1) AS seq_pct FROM pg_stat_user_tables ORDER BY seq_scan DESC LIMIT 10;

OOM / Memory Exhaustion

Memory diagnosis and mitigation
-- Identify high work_mem consumers (temp file usage) SELECT datname, temp_files, temp_bytes, pg_size_pretty(temp_bytes) AS temp_size FROM pg_stat_database ORDER BY temp_bytes DESC; -- Count connections using high memory (idle ones are wasteful) SELECT state, COUNT(*) FROM pg_stat_activity GROUP BY state; -- Emergency: kill all idle connections to reclaim memory SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle' AND usename != 'postgres' AND now() - state_change > interval '10 minutes'; -- Fix: reduce work_mem dynamically ALTER SYSTEM SET work_mem = '8MB'; SELECT pg_reload_conf();

Replication Lag Spike

Diagnose and resolve lag on read pool
-- On primary: check replication status SELECT application_name, state, sync_state, pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes, replay_lag FROM pg_stat_replication; -- Common causes of lag: 1. Heavy write transaction on primary (large bulk import) 2. Read pool CPU saturated (scale the pool) 3. Network partition (check Cloud Console VPC health) 4. Vacuuming & Table Bloat
AlloyDB uses ML-accelerated autovacuum but DBAs must still monitor bloat, XID wraparound risk, and tune autovacuum for high-churn tables.

Why Vacuum Matters

PostgreSQL uses MVCC — dead rows from UPDATE/DELETE are not immediately removed. Vacuum reclaims this space and prevents Transaction ID (XID) wraparound, which is a critical failure mode that causes the database to go read-only.

Autovacuum Monitoring

Tables at risk — bloat and XID age
-- Tables with most dead tuples (vacuum needed) SELECT schemaname, relname, n_dead_tup, n_live_tup, ROUND(100.0 * n_dead_tup / (n_live_tup + n_dead_tup + 1), 1) AS dead_pct, last_autovacuum, last_autoanalyze, autovacuum_count FROM pg_stat_user_tables WHERE n_dead_tup > 10000 ORDER BY n_dead_tup DESC LIMIT 20; -- XID wraparound risk (CRITICAL — alert if age > 1.5 billion) SELECT datname, age(datfrozenxid) AS xid_age, ROUND(age(datfrozenxid) / 2100000000.0 * 100, 2) AS pct_to_wraparound FROM pg_database ORDER BY xid_age DESC; -- Active autovacuum workers SELECT pid, relid::regclass AS table_name, phase, heap_blks_total, heap_blks_scanned, now() - query_start AS running_for FROM pg_stat_progress_vacuum;

Tuning Autovacuum for High-Churn Tables

Per-table autovacuum overrides
-- For high-write tables: vacuum more aggressively ALTER TABLE orders SET ( autovacuum_vacuum_scale_factor = 0.01, -- Vacuum at 1% dead rows (default 20%) autovacuum_analyze_scale_factor = 0.005, -- Analyze at 0.5% changes autovacuum_vacuum_cost_delay = 2, -- Less throttling (ms) autovacuum_vacuum_cost_limit = 800 -- More I/O budget (default 200) ); -- Manual vacuum (when autovacuum isn't keeping up) VACUUM (ANALYZE, VERBOSE) orders; -- FULL vacuum (reclaims disk space but LOCKS TABLE — use with caution) -- Better alternative: pg_repack extension (online, no lock) VACUUM FULL old_archive_table;
⚠️
VACUUM FULL takes an exclusive lock that blocks all reads and writes. For production tables, use pg_repack instead — it rebuilds the table online. For AlloyDB, bloat also increases storage costs.
Locks & Deadlocks
Lock contention is one of the top causes of production slowdowns. Learn to find, diagnose, and resolve lock conflicts quickly.

Finding Lock Conflicts

Complete blocking chain query
-- Full blocking chain (who is blocking whom) WITH lock_info AS ( SELECT blocked.pid AS blocked_pid, blocked.usename AS blocked_user, blocking.pid AS blocking_pid, blocking.usename AS blocking_user, blocked.query AS blocked_query, blocking.query AS blocking_query, now() - blocked.query_start AS blocked_duration, now() - blocking.query_start AS blocking_duration FROM pg_stat_activity blocked JOIN pg_locks blocked_locks ON blocked_locks.pid = blocked.pid JOIN pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype AND blocking_locks.relation = blocked_locks.relation AND blocking_locks.granted = true AND blocked_locks.granted = false JOIN pg_stat_activity blocking ON blocking.pid = blocking_locks.pid WHERE blocked_locks.pid != blocking_locks.pid ) SELECT * FROM lock_info ORDER BY blocked_duration DESC;

Deadlock Diagnosis

Find deadlocks in PostgreSQL logs
-- Deadlocks are logged automatically. Check Cloud Logging: -- resource.type="alloydb.googleapis.com/Instance" -- textPayload:"deadlock detected" -- Count deadlocks per database SELECT datname, deadlocks FROM pg_stat_database ORDER BY deadlocks DESC; -- Deadlock prevention: always acquire locks in the same order -- Example: always lock lower user_id first BEGIN; SELECT * FROM accounts WHERE id = LEAST(101, 202) FOR UPDATE; SELECT * FROM accounts WHERE id = GREATEST(101, 202) FOR UPDATE; COMMIT;

DDL Lock Hazards

Safe schema changes on live tables
-- DANGEROUS: takes ACCESS EXCLUSIVE lock → blocks all reads + writes ALTER TABLE orders ADD COLUMN notes TEXT; -- still instant in PG14+ (no rewrite) ALTER TABLE orders ADD COLUMN total NUMERIC -- RISKY: non-null with default = full rewrite NOT NULL DEFAULT 0; -- SAFE pattern: nullable first, backfill, then add constraint ALTER TABLE orders ADD COLUMN total NUMERIC; -- Step 1: instant UPDATE orders SET total = 0 WHERE total IS NULL; -- Step 2: batch ALTER TABLE orders -- Step 3: validate in background ADD CONSTRAINT total_not_null CHECK (total IS NOT NULL) NOT VALID; ALTER TABLE orders VALIDATE CONSTRAINT total_not_null; -- Step 4: ShareLock only -- Set statement timeout to avoid DDL blocking forever SET lock_timeout = '3s'; ALTER TABLE orders ADD COLUMN notes TEXT;
AI & pgvector
AlloyDB is Google's preferred vector database for AI workloads, with native pgvector support and ScaNN-based indexing for billion-scale vector similarity search.

Why AlloyDB for AI Workloads

🔢
pgvector (OSS)
Standard vector type and exact/approximate nearest-neighbor search. Up to 1536 dimensions (OpenAI embedding size). IVFFlat and HNSW indexes.
🚀
ScaNN Index (AlloyDB-native)
Google's ScaNN algorithm: 10x faster than IVFFlat on recall benchmarks at scale. Use for >1M vectors. Requires AlloyDB-specific extension.

Setting Up pgvector

Vector store for embeddings
-- Enable pgvector CREATE EXTENSION IF NOT EXISTS vector; -- Create a table with embedding column CREATE TABLE documents ( id BIGSERIAL PRIMARY KEY, content TEXT, metadata JSONB, embedding vector(1536) -- 1536 for text-embedding-ada-002 ); -- Insert an embedding (from your app) INSERT INTO documents (content, embedding) VALUES ('AlloyDB is a managed PostgreSQL service', '[0.012, -0.034, 0.198, ...]'::vector); -- Create HNSW index (best recall, recommended for <1M vectors) CREATE INDEX idx_documents_embedding ON documents USING hnsw (embedding vector_cosine_ops) WITH (m=16, ef_construction=64); -- Semantic similarity search (nearest 5 neighbors) SELECT id, content, embedding <=> '[0.021, -0.011, ...]'::vector AS distance FROM documents ORDER BY distance LIMIT 5;

ScaNN Index (AlloyDB Exclusive)

Enable and use ScaNN for large-scale search
-- Enable ScaNN CREATE EXTENSION IF NOT EXISTS alloydb_scann; -- Create ScaNN index (great for >1M vectors) CREATE INDEX idx_docs_scann ON documents USING scann (embedding cosine) WITH (num_leaves=500); -- Query (same syntax as pgvector) SET scann.num_leaves_to_search = 50; -- recall/speed tradeoff SELECT id, content, embedding <=> $1 AS dist FROM documents ORDER BY dist LIMIT 10;

pg_dump / pg_restore Migration

Zero-downtime migration pattern
# Step 1: Dump schema only first pg_dump --schema-only -Fc \ -h SOURCE_HOST -U postgres mydb > schema.dump # Step 2: Restore schema to AlloyDB pg_restore --schema-only -Fc \ -h ALLOYDB_IP -U postgres -d mydb schema.dump # Step 3: Dump and restore data (parallel for speed) pg_dump -Fd --jobs=8 \ -h SOURCE_HOST -U postgres mydb -f dump_dir/ pg_restore -Fd --jobs=8 \ -h ALLOYDB_IP -U postgres -d mydb dump_dir/ # Step 4: Set up logical replication to catch up delta # (On source): wal_level=logical, create publication CREATE PUBLICATION alloydb_migration FOR ALL TABLES; # (On AlloyDB): create subscription CREATE SUBSCRIPTION from_source CONNECTION 'host=SOURCE_IP dbname=mydb user=repl password=X' PUBLICATION alloydb_migration; # Step 5: Wait for lag = 0, then cutover app connection strings # Step 6: Drop subscription, enjoy AlloyDB
DBA Runbook & Checklists
Your production reference checklists — daily, weekly, and for common incident types.

Daily DBA Health Checks

Check replication lag

Run pg_stat_replication on primary. Read pool lag should be <30s. Alert if approaching 60s.

Review slow query log

Check Cloud Logging for queries >1000ms. Review Query Insights dashboard in Cloud Console.

Monitor connection counts

Ensure connections <80% of max_connections. Check for idle-in-transaction accumulation.

Verify backup completion

Confirm last automated backup succeeded in Cloud Console → AlloyDB → Backups.

Check XID age

Query pg_database for datfrozenxid age. Alert if >1.5 billion (danger zone is ~2.1B).

CPU and memory baseline

Review Cloud Monitoring dashboard. Spikes are normal; sustained >80% requires investigation.

Weekly DBA Tasks

  • Run ANALYZE on tables with high insert/update rates that autovacuum may miss
  • Review pg_stat_statements and reset after noting top-N queries
  • Check for bloated tables (n_dead_tup > 20% of n_live_tup)
  • Review unused indexes and schedule removal during low-traffic window
  • Validate that on-demand backups exist before any planned schema changes
  • Review IAM permissions — any service accounts with excessive privileges?
  • Check AlloyDB release notes for upcoming maintenance or version changes
